### User State and data to send

Right now, Cardsbolt uses ISR which doesn't change the props unless I ask for a revaliatePath(). Problem is, I have to show everyone the information before use. This means users would be able to get all the results via Web Developer Tools, even if in the UI it's blocking. There's no solution for this unless I go SSR, and that's too slow for what I want given the limitations I have. Therefore I'll have to move to the App Router. I am still to use the App Router but from what I can understand, I can cache function calls and I can use NextAuth to authenticate JWT tokens. I can use the JWT to confirm it hasn't been tampered with, therefore know what type of user they are, get the cached function and serve it to the user. As both are happening on the server now, I'm pretty sure this shouldn't impact on load times while the shell loads instantly. There's a bit of learning,

Basically would need to change the site to App Router which seems annoying. I'll get to that and I'll learn that properly as there's framework knowledge I need to know and am interested in, so we'll do that next week. As far as I can figure, it'll still load a shell and given the assets are cached also, all the results will be also, therefore the pages should still load instantly. I'll read up on this more and come back, but I'm fairly sure this strategy should work fine. Looking forward to giving it a shot in a branch.

### TCG Marketplace costs

If you want Cardsbolt to do the heavy lifting and find the individual product in question, it can do that. The other end of the stick is shops don't do a great job in selling from their websites, thus have to sell in an external marketplace like TCGPlayer or Cardsmarket. Now these shops usually have their own website and it would be cheaper to sell on it. Therefore I'm technically already serving them a good service because as long as their products are sold at a reasonable price, they'll get sold from the shop. They'll also not have to worry about PPC costs also because that gets expensive, though my idea might be too much to help but I'm still brainstorming this out. What I could do though is host all the products from the shop, be it cards, boxes, bundles, cases and so on, on my site. The difference is, if the page exists, I'll link to it. This eliminates the need of a middle man exchanging monies and also costs in relation to selling. People will go to the sites for the shops natively and buy from there, which should be cheaper while being the best experience. Now there's some technical hurdles because I don't want to be scrapping for millions of products so we have to go about it the Observer Listener pattern. Thankfully I've got a little bit of experience on this.

#### Shop adds code

This is a very very basic idea so stay with me. I give websites a certain API key which identifies who they are. Most likely an ID and that's about it. Whenever the shop updates the product on their site, which would be the price or available state, a PATCH call is fired to my API or an API. I'm not sure. In the patch, it'll have something in the body which gives the product SKU or SKU name. Then it'll give the price and availability. Really it can be one or another or both. With the shopId and productSku, I can then know which shopProduct to look at, which webpage to update and then update.

Now there's still the idea of setting this up where I'll need to look at their product inventory but again, they could send a POST with an array and because I have their shop, in very crude, if the product exists already in Cardsbolt, we populate the shopProduct. If not, the shop uses the productService and creates a product, it confirms the product is real via an API call to maybe OpenTCG (I would sponsor that service at this point), creates the product but only for that shop. Usually when I create a product, it's for every shop but in this case, the product is not a default product for Cardsbolt until defined (I'll need to add that oh boi). Now the product is made with the shop, we get the price and state and we're good to go once again.

That's the outline. I already grasp most of these shop owners aren't web developers so getting them to do this is a bit of a laugh. The only way I can make this work, is to create the plugins for Shopify, Woocommerce, Wix and other ecommerce platforms. They then just need to add the plugin with an API key and I do the rest. This also means I have to add some form of domain name to my API. There's always something. The idea though is with this plugin, it's able to look at the products, and pipe into the process of when a product is being updated, that I'm notified.
